# Full Admin System — Visitor Management System

## Context

The VMS has a solid backend (Express + Prisma + PostgreSQL) with JWT auth, 3 user roles (admin/manager/user), 24 granular permissions, and a complete API. However, none of this is wired up: no API routes are protected, no frontend auth guard exists, and there are no admin-specific pages. The login page stores a token in localStorage and redirects to `/dashboard` regardless of role. Every page duplicates its own header/nav.

## Goal

Build a complete admin system: auth guard middleware, shared authenticated layout with sidebar navigation, admin dashboard with role-based routing, user management (CRUD), and logout functionality.

---

## Phase 1: Backend — API Route Protection

### 1a. Add User Management API endpoints
**File:** `apps/api/src/routes/users.ts` (new)

- `GET /api/users` — List all users (admin only). Returns id, name, email, role, lastLogin, createdAt, organization.
- `GET /api/users/:id` — Get user by ID.
- `POST /api/users` — Create user (admin only). Validate with Zod, hash password, assign role.
- `PUT /api/users/:id` — Update user (admin only). Can update name, email, role. Optional password reset.
- `DELETE /api/users/:id` — Delete user (admin only). Prevent self-deletion.
- `POST /api/users/:id/reset-password` — Admin reset a user's password.

All endpoints protected with `authenticateToken` + `requireRole(["admin"])`.

### 1b. Protect existing API routes
**File:** `apps/api/src/index.ts` (edit)

Apply `authenticateToken` middleware to all API routes **except**:
- `POST /api/auth/login`
- `POST /api/auth/register`
- `GET /health`
- `GET /api`

Apply `requireRole(["admin"])` to sensitive routes:
- `DELETE /api/visitors/:id`
- `DELETE /api/employees/:id`
- `DELETE /api/sites/:id`
- All `/api/users/*` routes
- `GET /api/audit/*`

### 1c. Register users route
**File:** `apps/api/src/index.ts` (edit)

Add `import userRoutes from "./routes/users"` and `app.use("/api/users", userRoutes)`.

## Phase 2: Frontend — Auth Context & Middleware

### 2a. Auth context provider
**File:** `apps/web/src/lib/auth-context.tsx` (new)

- React Context providing: `user`, `token`, `login()`, `logout()`, `isAuthenticated`, `hasPermission()`.
- On mount: check localStorage for token, validate with `GET /api/auth/me`. If invalid/expired, clear and redirect to `/login`.
- Expose `useAuth()` hook.

### 2b. Root layout with AuthProvider
**File:** `apps/web/src/app/layout.tsx` (edit)

- Wrap `{children}` in `<AuthProvider>`.
- Keep existing Inter font and metadata.

### 2c. Next.js middleware for route protection
**File:** `apps/web/middleware.ts` (new)

- Protect all routes under `/dashboard`, `/admin`, `/visitors`, `/host`, `/reports`, `/audit`, `/badges`, `/areas`.
- Check for `token` in localStorage (via cookie or header — since Next middleware can't access localStorage, we'll use a lightweight approach: redirect to `/login` if no cookie, or rely on the AuthProvider for client-side protection).
- **Approach:** Since the entire app is client-side rendered, we'll use the AuthProvider as the primary guard. The Next.js middleware will only redirect `/` to `/login` if no auth cookie exists (optional, lightweight).

### 2d. API helper with token injection
**File:** `apps/web/src/lib/api.ts` (new)

- `apiClient` function that wraps `fetch` and auto-attaches `Authorization: Bearer <token>` from localStorage.
- Replaces all hardcoded `http://localhost:3001/api/...` calls across the app.
