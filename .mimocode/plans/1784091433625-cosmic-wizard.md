# Pre-Registered Visitor Badge System

## Overview
Authenticated employees can create pre-registered visits, generate QR code badges with 12-hour validity, and send them to visitors via server-side email. When the visitor arrives and scans the badge at the kiosk, the actual check-in time is recorded.

## Architecture

### Current State
- Visits can be pre-created via `POST /api/visits` (status: `pending`)
- Badges are only generated at check-in time (`POST /api/visits/checkin`) with 8-hour expiry
- `BadgeShare` component exists but only provides client-side sharing (mailto:, copy link)
- `BadgeDisplay` component renders a badge with QR code, countdown timer, visitor/host info
- Kiosk check-in flow: welcome â†’ phone â†’ details â†’ visitor type â†’ host â†’ confirm â†’ badge

### Target State
- New `/pre-register` page where employees create visits AND generate badges separately
- Badge generation creates a badge with 12-hour expiry for a pending/approved visit
- Server-side email sending with branded template + badge link
- Kiosk handles pre-registered badges: validates expiry, updates check-in time, generates visit badge

### Step 1: Backend â€” New endpoint `POST /api/visits/:id/generate-badge`

**File:** `apps/api/src/routes/visits.ts`

Add a new route that:
1. Finds the visit by ID (must be `pending` or `approved` status)

### Step 4: Backend â€” Update check-in to handle pre-registered badges

**File:** `apps/api/src/routes/visits.ts` (existing `POST /checkin` route)

The existing QR code lookup flow already works:
1. Visitor scans QR at kiosk
2. `POST /api/visits/checkin` with `qrCode` parameter
3. Finds badge by QR code, validates expiry
4. Updates visit status to `checked_in` and sets `checkInTime`
5. Generates a new badge with 8-hour expiry for the physical visit

**Change:** When a pre-registered badge is scanned:
- Keep the existing 12-hour badge as the pre-registration proof
